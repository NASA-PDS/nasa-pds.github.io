DOI Metadata Guidelines

Note

This page is under constructions. Contributions, comments, and improvements welcome in our Github Issues

  • DOI Metadata Guidelines
    • Citing Conventions
      • [New in Schema 4.4]

ROOT

This is the root of the submission file document, and thus required.

Note that the content of the <resource> element is defined under an xs:all group, so that the immediate child nodes can appear in any order. The order here reflects the order in which they are listed in the schema.

REQUIRED

This is the assigned DOI identifier and must begin with the DataCite prefix “10.”

Required attribute: identifierType. This must have a value of DOI. For example:

<identifier identifierType="DOI">10.12345/abcde</identifier>

REQUIRED

This is the author list or equivalent. Creators should appear in priority order. Both individuals and institutions/organizations can be credited as creators.

If the only list we have is a list of editors, then it goes here, but also indicate in the <contributors> section that each person in the list is a contributor with the role of Editor. This is not an optimal solution, but it will do for now.

REQUIRED, repeatable

This class contains the identifying information for one creator. It is repeated for each creator. Note that the metadata schema does not distinguish between “authors” and “editors”. If some of the creators listed are editors rather than authors, repeat their creator information in the <contributors> section and indicate a role of “Editor”.

Fill out as much of this class as possible with information that was apropos at the time of the publication. So, if you know given name, include it; if you know the institutional affiliation, include it; if you can find and confirm the correct ORCID, include it; etc. This all helps in linking the data set into the literature and the author’s network.

REQUIRED

The string representing the name as it should appear in citations. For personal names, the format is “Family, Given” (all one string). Full first names are preferred for the metadata, in order to help identify authors unambiguously. (It is also relatively easy to turn a complete name into an initial.) For organizational names, use the full, formal name of the organization.

This tag has one optional XML attribute, nameType, which takes one of these values:

Organizational Personal

Best practice is to always include this attribute for creatorName.

OPTIONAL

For personal names, this should contain the string corresponding to the given name in the <creatorName> element. This should not be present for organizational names (but that is not enforced by the DataCite schema).

OPTIONAL

For personal names, this should contain the string corresponding to the family name (surname, patronymic, etc.). This should not be present for organizational names (but that is not enforced by the DataCite schema).

Note: For PDS purposes, at least, we should consider where to associate suffixes (Jr., Sr., III, etc.). If DataCite, ADS, or the AAS journals have a convention, we should follow that.

OPTIONAL, repeatable

This attribute provides a formal identifier for an individual or organization - for example, an author’s ORCID, or an organizational DOI. Only public identifiers should appear here, of course. If there is more than one applicable identifier, this element may be repeated.

Required attribute: nameIdentifierScheme. The value should be the common name or acronym for the identifier given (like “ORCID”). Optional attribute: schemeURI. The value should be the URI of the defining organization or schema (”http://orcid.org”, e.g.).

Where it can be obtained directly from the creator, or where we can be certain about a creator’s ORCID, we should add it to the metadata.

OPTIONAL, repeatable

This attribute provides an organizational affiliation (as free-format text) for the creator. It may be repeated. Organizational “creators” may also have affiliations, if that makes sense.

Optional attribute: affiliationIdentifier. The value should be a formal, permanent, unique identifier for the affiliated organization, beyond the name that is the value of the <affiliation> element. For example, a Research Organization Registry (ROR) ID would be ideal here.

Optional attribute: affiliationIdentifierScheme. The (human-recognizable) name of the identifier system (e.g., “ROR” would be sufficient in the above case.

Optional attribute: schemeURI. The URI of the identifier scheme, typically a URL.

REQUIRED

This class lists names or titles for the resource being identified. At least one title must be provided. This is the title that will be used to format citations, and it is also the title that users will see returned by various search interfaces. It needs to make sense in a general, non-PDS search context, so avoid acronyms and assuming that users will know that, for example, “Deep Impact” is also the name of a spacecraft and NASA mission.

Remember the primary title has to make sense in the context of an ADS-type search. Users may not know that data sets are included in their result set. The title we put here will be used to formulate citations of the data set as well as helping users to identify data of potential interest. So avoid acronyms (no matter how obvious they seem now) unless they include the full name. For example, “DIF” is not good on its own, but “Deep Impact Flyby (DIF) Spacecraft” is very good - explicit and contains the acronym that knowledgeable users might search on as well.

The value must come from the following list:</p> <blockquote> <div><p>{| cellpadding=”10”</p> </div></blockquote> </div></blockquote> <p>|- | valign=”top” |</p> <ul class="simple"> <li><p>ARK</p></li> <li><p>arXiv</p></li> <li><p>bibcode</p></li> <li><p>DOI</p></li> <li><p>EAN13</p></li> <li><p>EISSN</p></li> </ul> <p>| valign=”top” |</p> <ul class="simple"> <li><p>Handle</p></li> <li><p>IGSN</p></li> <li><p>ISBN</p></li> <li><p>ISSN</p></li> <li><p>ISTC</p></li> <li><p>LISSN</p></li> </ul> <p>| valign=”top” |</p> <ul class="simple"> <li><p>LSID</p></li> <li><p>PMID</p></li> <li><p>PURL</p></li> <li><p>UPC</p></li> <li><p>URL</p></li> <li><p>URN</p></li> <li><p>w3id</p></li> </ul> <p>|}</p> <blockquote> <div><p>Do not include more than one <em><relatedIdentifier></em> for the same publication/work to be cited or related. For general use, <strong>DOI</strong> is the preferred identifier for all relationship types. For citation, <strong>bibcode</strong> is acceptable, but should only be used if there is no DOI available. When both exist, ADS (who issue <em>bibcodes</em>) have specifically requested that we only use the DOI in this list to cite the other work. <strong>ISBN</strong> and <strong>arXiv</strong> might also be available and could be used in cases where this is no DOI.</p> <p>Required attribute: <em>relationType</em>. The value must come from the following list:</p> <blockquote> <div><p>{| cellpadding=”10”</p> </div></blockquote> </div></blockquote> <p>|- | valign=”top” |</p> <ul class="simple"> <li><p>IsCitedBy</p></li> <li><p>Cites</p></li> <li><p>IsSupplementTo</p></li> <li><p>IsSupplementedBy</p></li> <li><p>IsContinuedBy</p></li> <li><p>Continues</p></li> <li><p>IsNewVersionOf</p></li> <li><p>IsPreviousVersionOf</p></li> <li><p>IsPartOf</p></li> <li><p>HasPart</p></li> <li><p><strong>[New in Schema 4.4]</strong> IsPublishedIn</p></li> </ul> <p>| valign=”top” |</p> <ul class="simple"> <li><p>IsReferencedBy</p></li> <li><p>References</p></li> <li><p>IsDocumentedBy</p></li> <li><p>Documents</p></li> <li><p>IsCompiledBy</p></li> <li><p>Compiles</p></li> <li><p>IsVariantFormOf</p></li> <li><p>IsOriginalFormOf</p></li> <li><p>IsIdenticalTo</p></li> <li><p>HasMetadata</p></li> <li><p>IsMetadataFor</p></li> </ul> <p>| valign=”top” |</p> <ul class="simple"> <li><p>Reviews</p></li> <li><p>IsReviewedBy</p></li> <li><p>IsDerivedFrom</p></li> <li><p>IsSourceOf</p></li> <li><p>Describes</p></li> <li><p>IsDescribedBy</p></li> <li><p>HasVersion</p></li> <li><p>IsVersionOf</p></li> <li><p>Requires</p></li> <li><p>IsRequiredby</p></li> <li><p>Obsoletes</p></li> <li><p>IsObsoletedBy</p></li> </ul> <p>|}</p> <blockquote> <div><p>Following is a summary of the relationships most likely to be useful in our context.</p> <p>{| class=”wikitable” style=”background-color:lime”</p> </div></blockquote> <p>! <em>relationType</em> !! Use for: |- | Cites/IsCitedBy || Standard citation of previous/subsequent work |- | IsVariantFormOf || PDS4 versions of PDS3 data sets, where the PDS3 data set has a DOI and <em>no substantive changes</em> (i.e., an external peer review was <em>not</em> required for the PDS4 version) have been made in migrating the data from PDS3 to PDS4. This should be included in the DOI metadata for both the PDS3 and PDS4 versions. |- | IsNewVersionOf/IsPreviousVersionOf || Datasets that are new/previous versions (that is, same ID but different version number) of datasets that also have DOIs; in this case both DOIs should have their metadata updated. Also, if a PDS3-to-PDS4 migration <em>did</em> require substantive changes to the data file, then this is the appropriate relationship between the two. |- | Obsoletes/IsObsoletedBy || When a dataset supersedes a dataset with a <em>different</em> ID (in the same ID system), use this relationship. |- | IsSupplementTo || Data that is the basis of or direct result of a journal article, but that was <em>not</em> published as part of the article - especially where the data set is mentioned in that publication |- | IsReferencedBy/References || Bibliographic references (as opposed to citations) |- | IsDerivedFrom/IsSourceOf || Raw-Calibrated relationships; high-order products and the contributing source products |- | IsPartOf/HasPart || Bundles and their constituent collections when both have DOIs. <em>IsPartOf</em> can also be used when a data product comprises one or more tables that were published (in their entirety) as part of a journal article or book. |- | colspan=”2” | Other relationships can and should be used where they seem useful. |}</p> <blockquote> <div><p>Optional attribute: <em>resourceTypeGeneral</em>. This attribute is identical to the one of the same name in <em><</em><a class="reference external" href="#.3CresourceType.3E">resourceType</a><em>></em>, above.</p> <p>These optional attributes should only be used when the value of the <em>relationType</em> attribute is either <strong>IsMetadataFor</strong> or <strong>HasMetadata</strong> (this is not validated):</p> <blockquote> <div><p>Optional attribute: <em>relatedMetadataScheme</em>. This indicates the ID or name of a metadata definition standard. Optional attribute: <em>schemeURI</em>. This should be the URI of the named metadata standard. Optional attribute: <em>schemeType</em>. The DataCite definition is not clear, but this looks like a specific file format type for the referenced metadata standard (such as “XSD”).</p> </div></blockquote> </div></blockquote> <section id="new-in-schema-4-4"> <span id="id2"></span><h3><strong>[New in Schema 4.4]</strong><a class="headerlink" href="#new-in-schema-4-4" title="Permalink to this headline"></a></h3> <p><em>OPTIONAL</em></p> <p>This element is used for two purposes:</p> <p>1. To provide reference information for a traditional publication that has no DOI; and 2. To provide additional information about the publication context of a reference that does have a DOI (for example, to document the book that contains a chapter in the citation list).</p> <p>SBN will generally be using this element for the first purpose. Note that <em>no verification</em> is done by DataCite for the publication information in this class even when a DOI is present, so type carefully and check it twice.</p> <p id="section-29"><em>OPTIONAL</em></p> <p>A single instance of “related item” information. Like <em><relatedIdentifier></em>, this element has several attributes.</p> <blockquote> <div><p>Required Attribute: <em>relatedItemType</em>. This must have a value from the same list as the <em>resourceTypeGeneral</em> attribute of the <a class="reference external" href="#.3CresourceType.3E"><resourceType></a> element above.</p> <p>Required Attribute: <em>relationType</em>. This must have a value from the same list as the <em>relationType</em> attribute of the <a class="reference external" href="#.3CrelatedIdentifier.3E"><relatedIdentifier></a> element above.</p> </div></blockquote> <p id="section-30"><em>OPTIONAL</em></p> <p>If the related item has a digital identifier, it should be specified here in the same manner as would be used for <a class="reference external" href="#.3CrelatedIdentifier.3E"><relatedIdentifier></a>, above.</p> <blockquote> <div><p>Optional Attribute: <em>relatedItemIdentifierType</em>. This must have a value from the same list as <a class="reference external" href="#.3CrelatedIdentifier.3E">relatedIdentifier</a>, above. Unlike <em><relatedIdentifier></em>, however, this attribute is optional.</p> <p>Optional Attributes: <em>relatedMetadataScheme</em>, <em>schemeURI</em>, and <em>schemeType</em>. These are used in the same circumstances and in the same way as for <a class="reference external" href="#.3CrelatedIdentifier.3E"><relatedIdentifier></a>, above. For SBN data sets, these are very unlikely to be relevant.</p> </div></blockquote> <p id="section-31"><em>OPTIONAL</em></p> <p>This is a grouping element for the creator list of the related item.</p> <p id="section-32"><em>OPTIONAL</em></p> <p>This element is similar to the <em><creator></em> element for the resource itself, but does not allow specification of a <em>nameIdentifier</em> or <em>affiliation</em>.</p> <p id="section-33"><em>REQUIRED</em></p> <p>As with the creators listed for the resource, the name should be given in the format “<em>familyName, givenName</em>”.</p> <blockquote> <div><p>Optional Attribute: <em>nameType</em>, which must be either <strong>Organizational</strong> or <strong>Personal</strong>.</p> <p>Optional Attribute: <em>xml:lang</em>, which can be used to indicate the language in which the name is expressed.</p> </div></blockquote> <p id="section-34"><em>OPTIONAL</em></p> <p>The given name of the creator.</p> <p id="section-35"><em>OPTIONAL</em></p> <p>The family name of the creator.</p> <p id="section-36"><em>OPTIONAL</em></p> <p>This container class is used to hold the title of the related item, and any alternate titles for that item.</p> <hr class="docutils" /> <title><hr class="docutils" /> <p><em>OPTIONAL</em></p> <p>This element holds the title of the related item, if it has one. It may be repeated if that seems appropriate.</p> <blockquote> <div><p>Optional Attribute: <em>titleType</em>. If there is more than one title, then the secondary titles should have an associated <em>titleType</em> to explain their existence. The values come from the same list as used for the <a class="reference external" href="#.3Ctitle.3E">title of the resource</a>, above.</p> </div></blockquote> <p>Note there is <strong>not</strong> an enclosing <em><titles></em> class in this case.</p> <p id="section-37"><em>OPTIONAL</em></p> <p>If the related item has aassociated volume identifier (numeric or otherwise), here’s where it goes.</p> <p id="section-38"><em>OPTIONAL</em></p> <p>If the related item has an associated issue identifier (numeric or otherwise), it goes here.</p> <p id="section-39"><em>OPTIONAL</em></p> <p>This element is for an associated sequence number, or similar (possibly non-numeric) identifier.</p> <blockquote> <div><p>Optional Attribute: <em>numberType</em>. Use the attribute to indicate what kind of number (corresponding to common prefixes in reference list formats) the element contains. It must have one of these values:</p> <blockquote> <div><p>{| cellpadding=”10”</p> </div></blockquote> </div></blockquote> <p>|- | valign=”top” |</p> <ul class="simple"> <li><p>Article</p></li> <li><p>Chapter</p></li> <li><p>Report</p></li> <li><p>Other</p></li> </ul> <p>|}</p> <p id="section-40"><em>OPTIONAL</em></p> <p>This and the following element are for specifying a start and end page for the item within the enclosing volume, issue, etc. If only one page is known, assume it is the starting page and use <em><firstPage></em> without <em><lastPage></em>.</p> <p id="section-41"><em>OPTIONAL</em></p> <p>This is for specifying the end of the page range of the related item within its containing publication. If only one page is known, assume it is the starting page and use <em><firstPage></em> without <em><lastPage></em>.</p> <p id="section-42"><em>OPTIONAL</em></p> <p>This element indicates the publisher of the related item, <em>not</em> the resource receiving the DOI.</p> <p id="section-43"><em>OPTIONAL</em></p> <p>The year in which the related item was made publicly available. As with the <em><publicationYear></em> of the resource, this must be in the range [0..9999].</p> <p id="section-44"><em>OPTIONAL</em></p> <p>This element is used to provide an edition or version number for the related item.</p> <p id="section-45"><em>OPTIONAL</em></p> <p>This is an enclosing element to hold a list of contributors to the related item.</p> <p id="section-46"><em>OPTIONAL</em></p> <p>This element identifies a single contributor.</p> <blockquote> <div><p>Required Attribute: <em>contributorType</em>. This indicates the role the named <em>contributor</em> filled. This must come from the same controlled-value list as listed above for the <a class="reference external" href="#.3Ccontributor.3E">contributors</a> to the resource.</p> </div></blockquote> <p id="section-47"><em>REQUIRED</em></p> <p>As with the contributors listed for the resource, the name should be given in the format “<em>familyName, givenName</em>”.</p> <blockquote> <div><p>Optional Attribute: <em>nameType</em>, which must be either <strong>Organizational</strong> or <strong>Personal</strong>.</p> <p>Optional Attribute: <em>xml:lang</em>, which can be used to indicate the language in which the name is expressed.</p> </div></blockquote> <p id="section-48"><em>OPTIONAL</em></p> <p>The given name of the contributor.</p> <p id="section-49"><em>OPTIONAL</em></p> <p>The family name of the contributor.</p> <p id="section-50"><em>OPTIONAL</em></p> <p>This element provides unstructured size information. In other words, it is not required to be numeric and there are no syntax constraints on the content.</p> <p id="section-51"><em>OPTIONAL</em></p> <p>A single size specification string, like “18GB” or “Three volumes”. This element may be repeated as needed or desired.</p> <table class="docutils align-default"> <colgroup> <col style="width: 100%" /> </colgroup> <thead> <tr class="row-odd"><th class="head"><p>Some Examples</p></th> </tr> </thead> <tbody> <tr class="row-even"><td><p><size>55GB</size></p></td> </tr> <tr class="row-odd"><td><p><size>3250 tables<size></p></td> </tr> <tr class="row-even"><td><p><size>235 FITS images<size></p></td> </tr> <tr class="row-odd"><td><p><size>24 documents in multiple formats<size></p></td> </tr> </tbody> </table> <p id="section-52"><em>OPTIONAL</em></p> <p>This class indicates the physical/digital format(s) of the resource.</p> <p id="section-53"><em>OPTIONAL</em></p> <p>This element contains a text description of the format. It is not constrained and may be repeated as appropriate.</p> <p>Best practice is to use a file extension or MIME type string as the value.</p> <table class="docutils align-default"> <colgroup> <col style="width: 100%" /> </colgroup> <tbody> <tr class="row-odd"><td><p>PDS should be more formal about the content here. This should be as explicit as possible, with a complete management statement where appropriate. Embargo information should also be recorded here.</p> <blockquote> <div><p>Optional attribute: <em>rightsURI</em>. This should be the URI to the full text of the license. Optional attribute: <em>rightsIdentifier</em>. A short, standardized license name. Best practice is to use one of the identifiers on the SPDX list: <a class="reference external" href="https://spdx.org/licenses/">https://spdx.org/licenses/</a> Optional attribute: <em>rightsIdentifierScheme</em>. Optional attribute: <em>xml:lang</em>. This indicates the language of the license.</p> </div></blockquote> <table class="docutils align-default"> <colgroup> <col style="width: 100%" /> </colgroup> <tbody> <tr class="row-odd"><td><p>PDS data is in the public domain for some values of “public domain”, but this is not a well-defined term. This field may be repeated as needed.</p> <p>It looks like formatting is not preserved for this text, but you may use the <em><br/></em> tag to insert a paragraph break.</p> <p>Best practice is to provide at least one description of some type. I don’t see a way to apply it as it currently exists to celestial coordinates, and that might seriously confuse applications that process this sort of metadata harvested from DOI databases. It’s included here for completeness.</p> <p id="section-60"><em>OPTIONAL</em></p> <p>This element defines one specific patch of Earth where the data were taken or on which the resource is focused. It may be repeated as desired.</p> <p id="section-61"><em>OPTIONAL</em></p> <p>A name for the location being defined.</p> <p>Note that the text provided in the schema that describes the points comprising the following elements as having values which are “a single latitude-longitude pair, separated by whitespace”. This is not true. In all cases there are tags specifically defining latitude and longitude as separate and distinct elements.</p> <p>It is also possible to define any single <em><geoLocation></em> as being simultaneously a single point, a single box, <em>and</em> any number of polygons. This seems irrational, yet it appears to be deliberate.</p> <p id="section-62"><em>OPTIONAL</em></p> <p>This element specifies a single point on the globe.</p> <p id="section-63"><em>REQUIRED</em></p> <p>Longitude in degrees in the range +/- 180.</p> <p id="section-64"><em>REQUIRED</em></p> <p>Latitude in degrees in the range +/- 90.</p> <p id="section-65"><em>OPTIONAL</em></p> <p>A box is defined by its four sides - east and west longtude, and north and south latitude.</p> <p id="section-66"><em>REQUIRED</em></p> <p>Westward bounding longitude in degrees in the range +/- 180.</p> <p id="section-67"><em>REQUIRED</em></p> <p>Eastward bounding longitude in degrees in the range +/- 180.</p> <p id="section-68"><em>REQUIRED</em></p> <p>Southward bounding latitude in degrees in the range +/- 90.</p> <p id="section-69"><em>REQUIRED</em></p> <p>Northward bounding latitude in degrees in the range +/- 90.</p> <p id="section-70"><em>OPTIONAL</em></p> <p>This element defines an arbitrary polygon as a sequence of points around the perimeter in which the last point must have the same definition as the first point (though this is not validated, nor is the nature of the path). There must be at least 4 points provided.</p> <p>Oddly, the schema allows this element to be repeated.</p> <p id="section-71"><em>REQUIRED</em></p> <p>Longitude and latitude of one point on the perimeter of the polygon.</p> <p id="section-72"><em>REQUIRED</em></p> <p>Longitude in degrees in the range +/- 180.</p> <p id="section-73"><em>REQUIRED</em></p> <p>Latitude in degrees in the range +/- 90.</p> <p id="section-74"><em>OPTIONAL</em></p> <p>If you are intending to define an area that is larger than half the total surface of the Earth, then you <em>must</em> use this element to define a point somewhere inside the area of interest. Otherwise the smaller enclosed area is assumed to be the area of interest. The actual point can be random, as long as it is inside the intended area.</p> <p id="section-75"><em>REQUIRED</em></p> <p>Longitude in degrees in the range +/- 180.</p> <p id="section-76"><em>REQUIRED</em></p> <p>Latitude in degrees in the range +/- 90.</p> <p id="section-77"><em>OPTIONAL</em></p> <p>This element identifies sources of funding related to creating or maintaining the resource.</p> <table class="docutils align-default"> <colgroup> <col style="width: 100%" /> </colgroup> <tbody> <tr class="row-odd"><td><p>This is not information that PDS has traditionally collected, but as it is used to trace results back to funding in the literature databases, we probably should.
